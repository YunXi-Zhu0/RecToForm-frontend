# RecToForm Frontend 项目理解与对接摘要

## 1. 项目目标

该前端项目基于 `Vite + React 19 + TypeScript`，目标是完成“发票识别到结果交付”的单页工作台闭环：

1. 初始化模板列表和标准字段
2. 切换处理模式
3. 上传多文件并创建异步识别任务
4. 轮询任务状态与进度
5. 获取识别结果 JSON 并渲染
6. 按模式下载或导出 Excel

当前后端支持两种业务模式：

- `template`：后端直接生成最终 Excel，前端负责模板选择、上传、轮询、预览、下载
- `standard_edit`：后端返回标准字段识别结果，前端负责表格编辑、修订、再导出

## 2. 当前仓库现状

从仓库现状看，工程基础已具备，但业务页面尚未开始：

- `@` 别名已配置，可直接按模块化目录组织代码
- `src/App.tsx` 仍是默认 Vite 示例页面，尚未替换为业务工作台
- `contexts/` 目录已存在但当前为空，适合沉淀后续实现上下文
- 目前最需要补齐的是业务类型、API 层、工作流层和工作台页面骨架

## 3. 核心业务闭环

推荐的主流程如下：

1. 页面初始化时并行请求 `templates` 与 `standard-fields`
2. 用户切换 `template` 或 `standard_edit`
3. 按模式准备上传配置
4. 调用 `POST /api/v1/tasks` 创建异步任务
5. 进入 2 到 3 秒一次的轮询
6. 任务结束后调用 `GET /api/v1/tasks/{task_id}/result`
7. `template` 模式直接下载结果文件
8. `standard_edit` 模式在前端编辑表格后再调用导出接口

## 4. API 与模式边界

### 4.1 初始化接口

- `GET /api/v1/templates`
- `GET /api/v1/templates/{template_id}`
- `GET /api/v1/standard-fields`

用途：

- 模板模式依赖模板列表和模板详情初始化只读列信息
- 自定义模式依赖标准字段全集初始化可编辑列

### 4.2 任务接口

- `POST /api/v1/tasks`
- `GET /api/v1/tasks/{task_id}`
- `GET /api/v1/tasks/{task_id}/result`

关键约束：

- 必须使用 `multipart/form-data`
- `config` 必须是 JSON 字符串
- 多文件字段名必须重复使用 `files`
- `template` 模式必须传 `template_id`
- `standard_edit` 模式不能传 `template_id`

### 4.3 下载与导出接口

- 模板模式：`GET /api/v1/tasks/{task_id}/excel`
- 自定义模式：`POST /api/v1/exports/standard-fields`
- 自定义导出下载：`GET /api/v1/exports/standard-fields/{export_id}`

关键边界：

- `template` 模式优先使用 `/result` 返回的 `excel_download_url`
- `standard_edit` 模式不能调用 `/tasks/{task_id}/excel`
- 自定义导出必须使用接口返回的 `download_url`，不要手拼地址

## 5. 结果渲染约束

### 5.1 模板模式

结果使用：

- `preview_headers`
- `preview_rows`
- `excel_download_url`
- `failed_items`

渲染规则：

- 第一列 `源文件` 或 `source file` 必须保持接口顺序
- 模板模式表格整体应为只读
- 下载按钮直接使用后端给出的下载地址

### 5.2 自定义表头模式

结果使用：

- `standard_fields`
- `rows`
- `failed_items`

渲染规则：

- 前端以 `standard_fields + rows` 初始化编辑态表格
- 前端负责改表头、改单元格、删列、调顺序
- 导出时需要从当前表格状态重新组装 `headers + rows`

## 6. 轮询、错误和边界处理

### 6.1 轮询规则

- 轮询间隔控制在 2 到 3 秒
- 当 `status` 为 `succeeded`、`partially_succeeded`、`failed` 时停止
- `status` 用于控制全局 UI
- `stage` 用于展示细粒度处理阶段
- 如果 `/result` 返回 `409 Conflict`，说明结果尚未就绪，应继续轮询状态接口

### 6.2 上传与重复文件

前端需做轻量校验：

- 文件数最多 `50`
- 单文件最大 `10MB`
- 仅支持图片和 PDF
- 本地可做弱重复提示，但以后端判定为准

服务端特殊错误：

```json
{
  "detail": {
    "message": "Duplicate files are not allowed.",
    "duplicate_files": ["a.png", "b.png"]
  }
}
```

前端处理要求：

- 不进入轮询态
- 保留用户当前选择文件
- 在对应文件项上展示重复状态

### 6.3 失败项与空值

- 任务允许 `partially_succeeded`
- `failed_items` 必须明确展示，不能吞掉
- 后端缺失值约定为 `""`
- 前端不能把空字符串擅自改成 `null`

## 7. 推荐状态模型

至少维护以下核心状态：

- `mode`
- `templates`
- `selectedTemplateId`
- `templateDetail`
- `standardFields`
- `uploadFiles`
- `duplicateFiles`
- `taskId`
- `taskStatus`
- `taskStage`
- `taskProgress`
- `taskResult`
- `failedItems`
- `editableHeaders`
- `editableRows`
- `exportId`
- `exportDownloadUrl`

## 8. 后续开发结论

当前最合理的落地顺序不是直接堆页面，而是先把类型、API 契约和工作流骨架立起来，再接 UI：

1. 先建立 `types / api / core / workflows` 的基础层
2. 再搭工作台页面与分区组件
3. 先完成 `template + standard_edit` 共用链路：初始化、上传、建任务、轮询、取结果
4. 再补 `standard_edit` 的编辑与导出
5. 最后补失败项展示、草稿缓存、体验增强

这能保证项目从一开始就按模块边界组织，避免把状态、接口和页面逻辑全部塞进 `App.tsx`
