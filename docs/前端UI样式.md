# RecToForm 前端 UI 样式与页面布局说明

## 1. 文档目标

本文档用于补充 RecToForm 前端页面的视觉方向、页面布局、核心组件和交互要求，为后续 UI 设计与前端实现提供统一参考。

整体目标不是做一个普通表单页，而是做成一个具备“智能识别工作台”气质的单页应用：清晰、专业、可编辑、具有科技感，同时适合持续扩展。

## 2. 整体视觉方向

### 2.1 设计关键词

- 科技感
- 智能识别工作台
- 深色氛围
- 数据面板感
- 精密、清晰、克制
- 有层次感但不过度花哨

### 2.2 建议视觉风格

推荐采用“深色未来感工作台”方向：

- 页面主背景采用深蓝黑或石墨黑基底
- 局部使用青蓝、冰蓝、荧光绿作为功能高亮色
- 通过微弱网格、径向光晕、模糊渐变、描边发光强化科技感
- 卡片、面板、表格区块需要有明确分层，而不是整页铺平
- 重点信息如任务状态、当前模式、下载动作要有强视觉锚点

不建议：

- 白底通用后台风格
- 过于廉价的紫色渐变科技风
- 到处都是圆角卡片的同质化设计

## 3. 页面整体布局

页面建议保持单页工作台结构，但视觉上拆成“头部 + 主工作区 + 辅助入口”。

### 3.1 顶部区域

顶部必须展示项目名称：

- `RecToForm`

建议同时包含以下内容：

- 项目标题
- 一句副标题，例如“Invoice Recognition Workbench”或“票据识别与结构化工作台”
- 当前模式状态标签
- 可选的系统状态提示位，例如“API Ready / Processing / Error”

顶部应具备品牌感，而不是普通表单标题。建议做成偏横向控制台抬头，包含轻微渐变描边或发光线条。

### 3.2 主工作区

主区域建议划分为三列或上下分层混合布局：

1. 左侧或顶部：模式与模板引导区
2. 中部：上传与任务进度区
3. 下部大区域：结果表格与下载导出区

推荐实际落地为：

- 第一屏：品牌头部 + 模式选择 + 模板引导 + 上传区 + 任务进度
- 第二屏或下半屏：结果表格编辑区 + 失败项区 + 下载导出区

这样可以保证用户先完成“输入与发起任务”，再进入“结果处理与导出”。

在具体实现上，建议进一步收敛成“单一共享工作区”模式：

- 页面中间放置一大块主工作区，作为整个任务流的核心操作舞台
- 上传文件、模板选择、任务分析、结果预览、表格编辑都在这块区域内完成
- 用户不是在多个分散模块间跳转，而是在同一块区域里根据当前步骤切换内容
- 这样更符合“工作台”体验，也更容易强化沉浸式科技感氛围

这个共享工作区应是页面视觉中心，面积要足够大，优先级高于其它辅助信息区。

### 3.3 左下角辅助入口

页面左下角预留一个“帮助中心”入口。

要求：

- 固定定位，始终可见
- 风格低调但可识别
- 支持后续跳转到独立说明页

建议表现：

- 左下角轻量悬浮按钮
- 文案：`帮助中心`
- 可配一个简洁图标，如问号、书本、指引箭头

## 4. 核心功能区块要求

### 4.0 中央共享工作区

中央共享工作区是本页面最核心的设计补充。

它不是简单的容器，而是整个业务流程的统一承载区。建议将用户主流程整理成 4 个连续状态：

1. 上传文件
2. 选择模板
3. 分析处理中
4. 结果预览与编辑

要求：

- 各阶段内容在同一块区域内切换
- 区域尺寸和结构尽量保持稳定，避免页面频繁跳动
- 切换时可加入淡入、模糊过渡、层叠切换等轻量动效
- 让用户明确感知“当前正在这个工作台中推进任务”，而不是跳转到多个独立页面

建议该区域具备明显边框、发光轮廓、轻微毛玻璃或深色半透明面板效果，形成沉浸式主舞台。

### 4.1 模式选择与模板引导区

该区域用于引导用户明确当前处理方式。

必须支持：

- `template` 模式
- 默认模板模式说明
- `standard_edit` 模式说明

推荐交互：

- 先展示两个预置模板作为首选方案
- 每个模板显示名称、用途、适用场景
- 如果用户不想使用预置模板，则可切换到“默认模板 / 全字段返回”模式
- 该默认模式本质上对应 `standard_edit`，由前端承接表头编辑

UI 上建议不要只用一个普通下拉框。更适合用：

- 模式切换卡片
- 模板选择卡片
- 配合当前激活态的发光描边或状态条

这样更有引导感，也更符合工作台氛围。

### 4.2 文件上传区

页面必须具备文件上传功能，并且需要承担较强的引导职责。

上传区建议支持：

- 点击上传
- 拖拽上传
- 选择文件夹上传
- 多文件列表展示
- 文件格式与大小提示
- 文件状态显示

推荐展示信息：

- 支持图片和 PDF
- 单文件不超过 `10MB`
- 单次最多 `50` 个文件
- 重复文件会被标记

文件列表每一项建议展示：

- 文件名
- 文件类型
- 文件大小
- 当前状态，例如“待上传 / 重复 / 已加入任务”

视觉上上传区可以做成发光边框拖拽框，强化“识别入口”的感觉。

在共享工作区的第一个状态中，页面应优先显示上传内容，不要让模板或结果抢占视觉焦点。

### 4.3 任务进度区

页面必须有实时任务进度追踪能力，进度条或轮盘都可以。

推荐做法：

- 同时提供总体进度条和状态文案
- 若视觉允许，可在旁边加入轮盘式阶段指示器

建议显示信息：

- 当前任务状态
- 当前处理阶段
- `processed / total`
- 百分比进度
- 成功数和失败数

状态区要具备明显反馈：

- `queued`：等待中
- `running`：处理中
- `partially_succeeded`：部分成功
- `succeeded`：已完成
- `failed`：失败

建议将进度区做成独立信息面板，避免和上传区混在一起。

如果采用共享工作区方案，则在“分析处理中”阶段，建议将主工作区整体做毛玻璃雾化处理：

- 底层保留上一阶段的模板或表格内容轮廓
- 上层加一层半透明雾化遮罩
- 中央放置轮盘式进度组件
- 实时显示进度百分比、当前阶段和简短状态文案

这样能让用户既知道系统正在处理，也不会失去对“当前任务对象”的上下文感知。

### 4.4 表格编辑区

前端表格编辑必须基于 `Handsontable`。

这里是页面的核心区域，应作为视觉主舞台。

必须支持：

- 在线编辑
- 复制
- 粘贴
- 只读预览与编辑模式区分

模式要求：

- `template` 模式下使用只读表格预览
- `standard_edit` 模式下使用可编辑表格

建议后续扩展能力：

- 改表头
- 批量粘贴
- 删列
- 调整列顺序

UI 建议：

- 表格区域占据页面最大面积
- 保持高信息密度，但表头、选中态、焦点态要清晰
- 表头与网格线风格应与整体科技风统一
- 可加入浅色高亮选择框，避免默认浏览器感

共享工作区中的表格阶段建议分成两个子阶段：

1. 模板选择阶段
2. 结果预览阶段

模板选择阶段建议：

- 用户上传完成后，在共享工作区下方或内部切换出模板选择区
- 模板区中可以直接使用 `Handsontable` 先展示一行表头预览
- 这一行表头用于展示“待填写表格长什么样”
- 用户可以基于这行表头确认自己选的是不是目标模板

结果预览阶段建议：

- 分析完成后，直接在同一共享工作区中展示完整结果表格
- `template` 模式下使用只读预览
- 自定义模板模式下可在结果区明确提示用户“现在可以自定义表头”
- 提示应出现在表格上方或工具栏位置，不要只靠 toast

### 4.5 下载与导出区

页面必须有表格下载按钮。

建议按模式区分按钮语义：

- `template` 模式：下载 Excel
- `standard_edit` 模式：导出并下载 Excel

按钮区建议包含：

- 主下载按钮
- 次级说明文字
- 可选任务结果摘要

下载按钮应是强主按钮，视觉层级明显，高于普通操作按钮。

在共享工作区最终阶段，下载按钮应与结果表格形成明确联动：

- 表格出现后，下载按钮进入主可用状态
- 若是自定义模板模式，可将按钮文案写为“导出表格”
- 若是预置模板模式，可将按钮文案写为“下载 Excel”

## 5. 单一共享工作区的交互流程

根据当前补充需求，页面主流程建议明确为以下状态机：

### 5.1 初始状态：上传文件

中央共享工作区默认展示上传界面。

功能：

- 拖拽上传
- 点击上传
- 选择文件夹上传
- 展示已选文件列表

目标：

- 用户先完成原始材料输入
- 上传是第一优先动作

### 5.2 文件上传后：模板选择

当用户上传了文件后，在共享工作区下切换出模板选择内容。

该阶段建议展示：

- 两个预置模板
- 默认模板 / 全字段返回选项
- 一行 `Handsontable` 表头预览
- 模板用途简述

并且必须预留返回能力：

- 页面左上角提供返回键
- 用户可回退到上传阶段补传文件或调整上传内容

这个返回键很重要，因为上传文件和模板确认存在强耦合，用户可能在看见表头后才发现漏传文件。

### 5.3 确认模板后：分析填表

用户确定模板后，按钮变为可点击状态，主按钮文案建议为：

- `分析填表`

点击后进入处理中状态。

该阶段视觉要求：

- 共享工作区整体毛玻璃雾化
- 中央叠加轮盘式进度展示
- 显示百分比走势
- 显示当前阶段文案
- 保留底层内容轮廓作为上下文

这样既能强化“系统正在计算”，也能保持页面连贯性。

### 5.4 分析完成后：结果预览与编辑

当分析完成后，共享工作区直接切换为结果表格视图。

该阶段要求：

- 展示填写完成后的表格
- 支持预览最终结果
- 如果用户之前选择的是自定义模板，则要明确提示“可以自定义表头了”
- 下载或导出按钮在这个阶段提供

这个阶段应让用户感到“任务已经完成，现在进入人工校对与交付阶段”。

## 7. 共享工作区的额外实现建议

为了让这块中部主舞台更完整，建议补充以下细节：

### 6.1 状态切换动效

- 上传完成后，模板区从下方滑入或渐显
- 点击“分析填表”后，主区进入模糊覆盖态
- 结果完成后，遮罩退场，表格内容淡入

### 6.2 左上角返回键

建议左上角返回键只在“模板选择”及后续阶段可见。

作用：

- 返回上传阶段
- 允许用户补传文件
- 减少错误流程下的挫败感

### 6.3 模板确认逻辑

建议在模板未确认时：

- “分析填表”按钮不可点击
- 通过视觉弱化提示用户先确认模板

### 6.4 自定义模板提示

如果用户选择的是默认模板，也就是全字段返回模式，则分析完成后建议在表格上方展示一条内嵌提示：

- 当前为全字段返回模式
- 你现在可以根据目标表结构自定义表头

这个提示比单纯依赖用户记忆更稳妥。

## 8. 科技感氛围的具体实现建议

为了让页面真正有“科技感”，建议在 UI 层做以下补充：

### 6.1 背景层

- 深色主背景
- 局部径向渐变光斑
- 极淡网格或坐标线纹理
- 控制整体对比度，避免背景喧宾夺主

### 6.2 面板层

- 面板使用半透明深色底
- 配合细描边和轻微内阴影
- 激活状态可出现蓝青色发光边框

### 6.3 字体层

- 标题使用更有识别度的展示字体
- 正文与数据区使用清晰稳定的无衬线字体
- 数值、进度、状态可用偏等宽风格增强仪表感

### 6.4 动效层

- 页面首次加载可加入分区渐进出现
- 模式切换和上传区 hover 可加入轻量位移或辉光
- 任务处理中可对进度区加入节奏性微动画

注意：

- 动效要服务于信息反馈
- 不要做成花哨的展示页

## 9. 建议补充的状态设计

为了让页面更完整，建议在 UI 设计里提前覆盖这些状态：

- 初始化加载态
- 模板加载失败态
- 标准字段加载失败态
- 未上传空态
- 已上传待选模板态
- 任务进行中态
- 结果为空态
- 部分成功态
- 导出中态
- 下载完成态

这部分虽然不是你本次显式提出的，但对真正可用的工作台是必要补充。

## 10. 响应式建议

虽然主场景以桌面端为主，但仍建议保证中等屏幕可用：

- 桌面端优先采用宽屏工作台布局
- 窄屏时将上传区和进度区改为上下堆叠
- 表格区保持横向滚动，不强行压缩列宽
- 左下角帮助中心在移动端可改为右下角浮动入口

## 11. 最终设计结论

RecToForm 的页面不应是普通后台表单，而应是一个带有“识别控制台”气质的智能工作台。

本次 UI 设计要求可以总结为：

1. 顶部必须突出 `RecToForm` 项目名称和品牌感
2. 中部需要以“单一共享工作区”承载上传、模板选择、分析、结果预览四个阶段
3. 上传区需要具备拖拽上传、文件夹上传、列表、状态展示能力
4. 模板确认后需要通过“分析填表”按钮进入处理中状态
5. 处理中需要使用毛玻璃雾化 + 轮盘百分比进度作为核心反馈
6. 核心结果区必须使用 `Handsontable` 承担在线编辑、复制、粘贴
7. 若为默认模板模式，分析完成后要明确提示用户可以自定义表头
8. 页面需要清晰的 Excel 下载或导出按钮
9. 页面左上角需要预留返回键，左下角需要预留“帮助中心”入口
10. 整体视觉必须围绕“深色、精密、数据工作台、科技感”展开

后续如果进入 UI 实现阶段，应以此文档作为页面布局和视觉基线。
