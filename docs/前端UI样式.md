# RecToForm 前端 UI 样式说明

## 1. 文档目标

本文档用于统一 RecToForm 当前前端页面的视觉方向、布局原则和关键交互要求。

当前页面不再走“科技感工作台”方向，而是明确采用：

- 轻松
- 简约
- 留白充足
- 排版合理
- 不冗余
- 像日常办公工具一样自然好用

目标不是做一个炫技页面，而是做一个清爽、安静、可持续扩展的单页工作台。

## 2. 最新视觉方向

### 2.1 关键词

- 浅色
- 轻松
- 简洁
- 柔和
- 纸面感
- 办公工具感
- 清晰但不过度强调“系统感”

### 2.2 明确要求

- 不要科技感
- 不要深色未来风
- 不要控制台氛围
- 不要过强发光、炫光、网格、仪表盘式压迫感
- 不要页面一上来堆很多面板和说明

### 2.3 推荐表现

- 使用浅色暖底或中性浅底
- 组件边框柔和、阴影轻量
- 字体稳重自然，避免太强的工业感或控制台感
- 用留白、层级和排版解决信息组织，而不是用强装饰堆砌
- 重点区域只保留一个视觉中心

## 3. 页面布局要求

页面必须保持单页结构，但首屏需要明显收敛。

### 3.1 页头

页面顶部必须展示品牌：

- `RecToForm`

要求：

- 品牌位于页面上方
- 页头简洁，不要占据太大高度
- 可以有一句轻量说明，但不能喧宾夺主

### 3.2 中部主区域

页面中间必须只有一个主要操作舞台。

要求：

- 页面中部有一个大的上传框
- 该区域支持拖拽上传
- 也支持点击选择文件
- 上传框是首屏最核心的视觉中心
- 首屏不要同时把模板区、结果区、失败区、导出区全部堆出来

### 3.3 同区域连续流程

上传文件后，不要跳到别的面板去做下一步，而是继续在同一块区域完成配置。

要求：

1. 先显示大上传框
2. 用户上传文件
3. 上传完成后，在同一块区域内要求用户选择处理模式
4. 如果是模板模式，再在同一区域补充模板选择
5. 再由用户启动任务

这条要求的核心是：

- 同一块区域承接上传、模式选择和任务启动
- 不要让用户在页面不同角落来回找下一步

## 4. 处理中态要求

任务开始处理后，主区域需要进入明显但轻量的处理中态。

要求：

- 虚化背景
- 在原区域上覆盖半透明遮罩
- 中央显示进度信息
- 有轮盘式进度组件
- 保留简洁的状态说明

建议表现：

- 底层仍能隐约看到刚才上传和配置的内容
- 上层只突出轮盘、进度百分比、阶段文案
- 避免复杂仪表盘和过度数据化布局

## 5. 首屏信息密度要求

首屏必须进一步简化。

建议首屏只保留：

- 页头品牌 `RecToForm`
- 一句简短说明
- 大上传框
- 上传后出现的处理模式选择
- 启动任务按钮

以下内容不应在无结果时抢占首屏：

- 大片结果表格
- 失败项筛选区
- 导出面板
- 过多系统状态块
- 冗长说明文案

## 6. 模块表现建议

### 6.1 上传区

上传区需要做到：

- 大尺寸
- 拖拽友好
- 空态清晰
- 文件限制信息简洁展示
- 文件列表不要过于厚重

### 6.2 模式切换

模式切换建议用简洁卡片或分段选择方式展示。

要求：

- 用户上传后才出现
- 文案简短
- 两种模式差异要能看懂
- 不要复杂解释

### 6.3 进度区

进度区在处理中才需要成为主视觉。

要求：

- 轮盘要有，但整体风格要轻
- 可以有百分比
- 可以有阶段文案
- 不要过强科技感

### 6.4 结果区

结果区应在任务完成后再出现，作为后置区域承接：

- 模板模式结果预览
- 标准字段可编辑表格
- 导出操作
- 失败项展示

要求：

- 首屏不抢视觉中心
- 完成任务后再自然展开

## 7. 响应式要求

- 桌面端优先
- 移动端下主上传框、模式卡、按钮要能自然折行
- 不允许因为大上传区或覆盖层导致内容溢出
- 处理中遮罩在移动端仍需保持居中和清晰

## 8. 当前应遵守的 UI 结论

当前 RecToForm UI 的明确方向是：

- 一个简洁页头
- 一个大的中部上传框
- 上传后在同一区域继续选择处理模式
- 提交后在该区域上方做虚化进度覆盖
- 整体采用轻松、简约、自然的浅色主题

只要后续继续改 UI，就必须以这组要求为准，不再回退到深色科技感方案。
