# 前端设计初步

## 1. 文档目标

本文档面向前端开发，整理当前可用 API、请求响应示例、`curl` 调用示例，以及前端页面组织、状态管理和联调注意事项。

当前系统围绕两种模式工作：

1. `template`
   后端按模板直接生成最终 Excel，前端主要负责上传、轮询、预览和下载。
2. `standard_edit`
   后端返回标准字段全集识别结果，前端负责结果表格编辑、修订和二次导出。

---

## 2. 整体交互闭环

主链路：

1. 前端加载初始化数据
2. 用户选择模式
3. 用户上传发票文件并创建异步任务
4. 前端轮询任务状态
5. 任务完成后按模式拉取结果
6. 模板模式直接下载 Excel
7. 自定义表头模式在前端编辑表格后调用导出接口

建议页面采用单页三段式工作台：

1. 顶部模式与方案区
2. 中部上传与任务进度区
3. 底部结果预览与导出区

---

## 3. API 总览

基础信息：

- 服务前缀：`/api/v1`
- 健康检查：`GET /health`
- 默认开启 CORS
- 长耗时识别走异步任务，不是同步返回

接口清单：

1. `GET /health`
2. `GET /api/v1/templates`
3. `GET /api/v1/templates/{template_id}`
4. `GET /api/v1/standard-fields`
5. `POST /api/v1/tasks`
6. `GET /api/v1/tasks/{task_id}`
7. `GET /api/v1/tasks/{task_id}/result`
8. `GET /api/v1/tasks/{task_id}/excel`
9. `POST /api/v1/exports/standard-fields`
10. `GET /api/v1/exports/standard-fields/{export_id}`

### 4.5 创建任务

功能：

- 上传多个文件
- 创建异步识别任务
- 立即返回 `task_id`

请求格式：

- `multipart/form-data`
- 一个 `config` 字段
- 多个 `files` 字段

`config` 结构：

```json
{
  "mode": "template",
  "template_id": "finance_invoice",
  "template_version": "v1",
  "extra_instructions": ["保留原票面日期格式"]
}
```

或：

```json
{
  "mode": "standard_edit",
  "extra_instructions": ["如缺失则返回空字符串"]
}
```

请求示例：

```http
POST /api/v1/tasks
Content-Type: multipart/form-data
```

响应示例：

```json
{
  "task_id": "7f7b5d9d9bdc4a35b8332ab84d9b6bf2",
  "status": "queued",
  "mode": "template",
  "total_files": 2,
  "duplicate_files": []
}
```

`curl` 示例，模板模式：

```bash
curl -X POST http://127.0.0.1:8080/api/v1/tasks \
  -F 'config={"mode":"template","template_id":"finance_invoice","template_version":"v1","extra_instructions":["保留原票面日期格式"]}' \
  -F 'files=@tests/fixtures/invoices/Snipaste_2026-03-18_01-04-58.png' \
  -F 'files=@tests/fixtures/invoices/汽油250808.pdf'
```

`curl` 示例，自定义表头模式：

```bash
curl -X POST http://127.0.0.1:8080/api/v1/tasks \
  -F 'config={"mode":"standard_edit","extra_instructions":[]}' \
  -F 'files=@tests/fixtures/invoices/Snipaste_2026-03-18_01-04-58.png'
```

常见错误：

```json
{
  "detail": "Unsupported mode: xxx"
}
```

```json
{
  "detail": "template mode requires template_id."
}
```

```json
{
  "detail": "Uploaded file exceeds 10MB: invoice.pdf"
}
```

```json
{
  "detail": "Unsupported file type: invoice.txt"
}
```

```json
{
  "detail": {
    "message": "Duplicate files are not allowed.",
    "duplicate_files": ["invoice-a.png", "invoice-a-copy.png"]
  }
}
```

前端注意：

- `config` 必须是 JSON 字符串，不是 multipart 的嵌套对象
- 多文件字段名是 `files`
- 不要在创建任务后等待同步完成，应直接进入轮询
- 成功响应里的 `duplicate_files` 当前固定返回空数组，前端可据此统一处理上传提示
- 如果服务端返回 `detail.duplicate_files`，应高亮这些文件名并阻止进入轮询态
- 重复判定基于文件内容，不依赖文件名；不同文件名但内容一致也会被拒绝

### 4.6 查询任务状态

功能：

- 轮询任务进度
- 驱动前端状态、进度条、成功失败统计

请求：

```http
GET /api/v1/tasks/{task_id}
```

响应示例：

```json
{
  "task_id": "7f7b5d9d9bdc4a35b8332ab84d9b6bf2",
  "mode": "template",
  "status": "running",
  "stage": "llm_processing",
  "total_files": 2,
  "processed_files": 1,
  "succeeded_files": 1,
  "failed_files": 0,
  "progress_percent": 50,
  "error_message": ""
}
```

`curl` 示例：

```bash
curl http://127.0.0.1:8080/api/v1/tasks/7f7b5d9d9bdc4a35b8332ab84d9b6bf2
```

当前可见状态和阶段值：

- `queued`
- `running`
- `file_preprocessing`
- `llm_processing`
- `assembling_results`
- `excel_generating`
- `succeeded`
- `partially_succeeded`
- `failed`

前端建议：

- 轮询间隔控制在 2 到 3 秒
- 当 `status` 进入 `succeeded`、`partially_succeeded`、`failed` 时停止轮询
- `stage` 用于展示当前处理环节，`status` 用于控制全局 UI

### 4.7 获取任务结果

功能：

- 在任务完成后获取可直接渲染的数据
- 不需要解析 Excel 文件

请求：

```http
GET /api/v1/tasks/{task_id}/result
```

如果任务还没完成，返回：

```json
{
  "detail": "Task result is not ready yet."
}
```

状态码：

- `409 Conflict`

#### 模板模式响应示例

```json
{
  "task_id": "7f7b5d9d9bdc4a35b8332ab84d9b6bf2",
  "mode": "template",
  "status": "partially_succeeded",
  "preview_headers": ["源文件", "序号", "发票号码", "发票代码", "发票金额", "备注"],
  "preview_rows": [
    ["invoice-a.png", "1", "INV-001", "CODE-001", "88.00", "A"],
    ["invoice-b.pdf", "2", "INV-002", "CODE-002", "99.00", "B"]
  ],
  "excel_download_url": "http://127.0.0.1:8080/api/v1/tasks/7f7b5d9d9bdc4a35b8332ab84d9b6bf2/excel",
  "failed_items": [
    {
      "file_id": "file-003",
      "file_name": "bad.pdf",
      "error_message": "parse failed"
    }
  ]
}
```

#### 自定义表头模式响应示例

```json
{
  "task_id": "7f7b5d9d9bdc4a35b8332ab84d9b6bf2",
  "mode": "standard_edit",
  "status": "partially_succeeded",
  "standard_fields": ["发票代码", "发票号码", "开票日期", "购买方名称", "备注"],
  "rows": [
    ["CODE-001", "INV-001", "2026-03-01", "甲公司", ""],
    ["CODE-002", "INV-002", "2026-03-05", "乙公司", "手工补充"]
  ],
  "failed_items": [
    {
      "file_id": "file-002",
      "file_name": "b.png",
      "error_message": "parse failed"
    }
  ]
}
```

`curl` 示例：

```bash
curl http://127.0.0.1:8080/api/v1/tasks/7f7b5d9d9bdc4a35b8332ab84d9b6bf2/result
```

前端注意：

- 模板模式使用 `preview_headers + preview_rows`
- 自定义表头模式使用 `standard_fields + rows`
- `failed_items` 必须展示，不能静默吞掉失败文件
- 模板模式第一列固定为 `源文件`，应保持只读，用于回溯该行结果来自哪个上传文件
- `源文件` 列已经进入后端导出的模板任务汇总 Excel，前端预览列顺序应与接口返回保持一致

### 4.9 导出自定义表头结果

功能：

- 接收前端编辑后的二维表
- 生成 Excel 导出文件
- 返回下载标识和下载地址

请求：

```http
POST /api/v1/exports/standard-fields
Content-Type: application/json
```

请求体示例：

```json
{
  "headers": ["发票号码", "发票代码", "合计", "备注"],
  "rows": [
    ["12345678", "011002300111", "100.00", ""],
    ["87654321", "011002300222", "230.00", "前端补充"]
  ],
  "filename": "发票信息.xlsx"
}
```

响应示例：

```json
{
  "export_id": "5d662518-2ff9-473d-bf2d-d4bb6fa7d1d3",
  "filename": "发票信息.xlsx",
  "download_url": "http://127.0.0.1:8080/api/v1/exports/standard-fields/5d662518-2ff9-473d-bf2d-d4bb6fa7d1d3"
}
```

`curl` 示例：

```bash
curl -X POST http://127.0.0.1:8080/api/v1/exports/standard-fields \
  -H 'Content-Type: application/json' \
  -d '{
    "headers":["发票号码","发票代码","合计","备注"],
    "rows":[
      ["12345678","011002300111","100.00",""],
      ["87654321","011002300222","230.00","前端补充"]
    ],
    "filename":"发票信息.xlsx"
  }'
```

校验要求：

- `headers` 不能为空
- `headers` 中不能有空白值
- `rows` 不能为空
- 每一行长度必须与 `headers.length` 一致
- `filename` 可省略，默认是 `standard_fields_export.xlsx`

前端注意：

- 这个接口不会直接返回文件流，而是返回下载地址
- 真实文件名不会出现在下载 URL 中，URL 中使用的是 `export_id`

## 5. 前端开发思路

### 5.1 页面结构建议

建议拆成以下区域：

1. 模式切换区
2. 模板或标准字段初始化区
3. 文件上传区
4. 任务进度区
5. 结果表格区
6. 导出与下载操作区

推荐的用户流程：

1. 进入页面后先请求模板列表和标准字段列表
2. 用户切换模式
3. 模板模式下选择模板，自定义表头模式下初始化可编辑空表
4. 上传文件并创建任务
5. 轮询状态直到结束
6. 拉取结果并渲染
7. 模板模式点击下载，自定义表头模式编辑后导出

### 5.2 建议的状态模型

建议至少维护以下状态：

- `mode`
- `templates`
- `selectedTemplateId`
- `templateDetail`
- `standardFields`
- `uploadFiles`
- `duplicateFiles`
- `taskId`
- `taskStatus`
- `taskStage`
- `taskProgress`
- `taskResult`
- `failedItems`
- `editableHeaders`
- `editableRows`
- `exportId`
- `exportDownloadUrl`

### 5.3 模板模式开发思路

模板模式重点是“任务驱动 + 只读预览 + 最终下载”：

1. 拉取模板列表
2. 用户选择模板并拉取模板详情
3. 创建任务时提交 `mode=template`
4. 轮询任务状态
5. 完成后调用 `/tasks/{task_id}/result`
6. 用 `preview_headers` 和 `preview_rows` 渲染预览，并将 `源文件` 列固定展示
7. 点击下载时跳转 `excel_download_url`

### 5.4 自定义表头模式开发思路

自定义表头模式重点是“后端返回全集，前端完成最终表格”：

1. 拉取标准字段全集
2. 使用 `fields` 初始化表头
3. 创建任务时提交 `mode=standard_edit`
4. 轮询任务状态
5. 完成后调用 `/tasks/{task_id}/result`
6. 把 `standard_fields` 作为初始列，把 `rows` 作为初始数据
7. 用户可改表头、改单元格、删列、调整顺序
8. 导出时将当前表格内容重新组装为 `headers + rows`
9. 调用 `/exports/standard-fields`
10. 收到 `download_url` 后再触发下载

---

## 6. 表格组件建议

建议使用 `Handsontable` 风格组件或具备以下能力的二维表格组件：

1. 可编辑单元格
2. 支持复制粘贴
3. 支持列拖拽排序
4. 支持删列
5. 支持批量粘贴 Excel 内容

组件使用建议：

- 模板模式可复用同一表格组件，但切换为只读态
- 自定义表头模式建议把“表头”和“内容”都纳入同一张可编辑表
- 导出前必须从当前前端表格状态重新生成 `headers` 和 `rows`

---

## 7. 联调与实现注意事项

### 7.1 任务轮询

- 不要把任务创建和结果获取写成一个同步请求
- 在 `queued`、`running` 状态下持续轮询
- 如果 `/result` 返回 `409`，说明结果尚未落盘，继续轮询状态接口

### 7.2 模式边界

- `template` 模式必须提供 `template_id`
- `standard_edit` 模式不能传 `template_id`
- `standard_edit` 模式不走 `/tasks/{task_id}/excel`

### 7.3 文件限制

当前后端限制来自配置：

- 最多 `50` 个文件
- 单文件最大 `10MB`
- 文件类型需被后端识别，当前支持图片和 PDF，未知类型会报错
- 同一次上传内禁止重复文件，后端会返回重复文件名列表

前端应在上传前做一次本地校验，但仍以服务端校验为准。

推荐的上传交互：

- 选中文件后先在本地按文件名和大小做一次弱提示，减少明显误传
- 仍以服务端 `detail.duplicate_files` 作为最终判定结果
- 如果后端返回重复文件，保留用户当前已选文件列表，并在文件项旁展示“重复上传”状态

### 7.4 空值处理

- 标准字段接口中的 `default_missing_value` 当前是空字符串 `""`
- 前端表格中看到空字符串不代表字段缺失逻辑异常，可能只是模型按约定返回空值
- 不要擅自把空字符串替换成 `null`

### 7.5 失败文件展示

- 任务可能 `partially_succeeded`
- `failed_items` 需要单独渲染
- 建议在结果区展示“成功数 / 失败数 / 失败文件列表”

### 7.6 下载处理

- 模板模式下载地址来自 `excel_download_url`
- 自定义表头模式下载地址来自导出接口响应的 `download_url`
- 自定义表头导出下载 URL 使用 `export_id`，不是文件名

### 7.7 错误处理

后端错误响应统一形态通常为：

```json
{
  "detail": "错误说明"
}
```

重复上传是例外，错误响应会返回结构化对象：

```json
{
  "detail": {
    "message": "Duplicate files are not allowed.",
    "duplicate_files": ["invoice-a.png", "invoice-a-copy.png"]
  }
}
```

前端应统一封装错误提示逻辑，避免每个接口各自处理。

### 7.8 不要依赖解析 xlsx

- 前端展示结果应依赖 `/result` 返回的 JSON
- 不要为了预览再去读下载下来的 Excel
- Excel 仅作为最终交付物

## 8. 推荐的前端实现优先级

第一阶段：

1. 模式切换
2. 模板和标准字段初始化
3. 多文件上传
4. 任务创建与轮询
5. 结果预览

第二阶段：

1. 自定义表头编辑
2. 复制粘贴
3. 删列和调列顺序
4. 自定义结果导出

第三阶段：

1. 失败项筛选展示
2. 表格草稿缓存
3. 导出文件名自定义
4. 结果恢复为初始识别值

---

## 9. 前后端联调建议

联调顺序建议如下：

1. 先打通 `GET /templates`、`GET /standard-fields`
2. 再打通 `POST /tasks` 和 `GET /tasks/{task_id}`
3. 再接 `GET /tasks/{task_id}/result`
4. 模板模式最后接 `GET /tasks/{task_id}/excel`
5. 自定义表头模式最后接 `POST /exports/standard-fields` 和下载接口

这样可以先保证“上传和结果回显”通路稳定，再补最终导出能力。
