# RecToForm Frontend 模块化实施计划

## 1. 目标

将当前默认 Vite 页面改造成“单页工作台”，并按工程化边界拆分为可持续迭代的业务前端。

## 3. 模块职责拆分

### 3.1 `types`

职责：

- 严格对齐后端字段命名
- 区分两种模式的结果结构
- 为 UI 状态和工作流状态提供明确类型边界

建议优先定义：

- 模板列表、模板详情类型
- 标准字段类型
- 创建任务请求与响应类型
- 任务状态响应类型
- 两种结果响应类型
- 导出请求与响应类型

### 3.2 `api`

职责：

- 只做 HTTP 请求与响应解析
- 不在这里写 UI 逻辑
- 每个资源一份模块，避免大而杂

拆分建议：

- `templates.ts`：模板列表、模板详情
- `standard-fields.ts`：标准字段初始化
- `tasks.ts`：创建任务、查询任务、获取结果、模板下载
- `exports.ts`：自定义导出与下载地址获取

### 3.3 `core`

职责：

- 全局常量
- API 基础配置
- 错误解析
- 上传校验
- 展示文案映射

建议优先落地：

- 模式常量、状态常量、阶段常量
- 文件大小和数量限制
- 后端错误 `detail` 统一解析器
- 重复文件错误识别器
- 任务状态到 UI 文案映射器

### 3.4 `workflows`

职责：

- 串联跨接口的业务流程
- 管理“建任务 -> 轮询 -> 拉结果 -> 更新状态”的流程化逻辑
- 将页面层从接口细节中解耦

优先工作流：

- `initialize-workbench-flow`
- `create-task-flow`
- `resolve-task-result-flow`
- `export-standard-fields-flow`

### 3.5 `hooks`

职责：

- 封装有状态但可复用的前端行为
- 让页面层更薄

建议优先落地：

- `use-workbench-state`
- `use-upload-files`
- `use-task-polling`

### 3.6 `components`

职责：

- 只做可组合 UI 区块
- 尽量通过 props 接收数据和回调，不直接感知 API

建议先做这些区块：

- 模式切换区
- 模板信息区
- 上传区
- 进度区
- 结果表格区
- 失败项区
- 下载导出操作区

### 3.7 `pages`

职责：

- 只负责组装工作台页面
- 把 `App.tsx` 保持为轻入口

## 4. 分阶段实施

### Phase 1：打通主链路

目标：

- 页面初始化
- 模式切换
- 多文件上传
- 创建任务
- 任务轮询
- 结果预览

交付标准：

- 两种模式都能完成上传到结果展示
- 模板模式能拿到下载地址
- `failed_items` 能显示

### Phase 2：补齐自定义编辑与导出

目标：

- 可编辑二维表
- 改表头、改单元格
- 删列、调顺序
- 调用导出接口并下载

交付标准：

- 可以从当前表格状态正确导出 Excel
- 不依赖后端原始顺序才能下载

### Phase 3：增强稳定性与体验

目标：

- 失败项过滤与聚合展示
- 草稿缓存
- 自定义导出文件名
- 恢复原始识别值

交付标准：

- 用户可以稳定恢复和继续编辑中间态
- 局部失败与重试信息清晰可见

## 6. 工程原则

- 内部模块统一使用 `@/...` 导入
- API 类型保持和后端字段完全对齐，不擅自改名
- 不把业务逻辑塞进 `App.tsx`
- 不混淆 `template` 与 `standard_edit` 的流程边界
- 不手工拼接下载地址
- 不解析下载下来的 Excel 做预览
- 不把空字符串转成 `null`

## 7. 风险点

- 自定义表格组件选型会影响后续复制粘贴、调列和性能
- `result` 接口可能在任务结束后短暂返回 `409`，轮询收敛逻辑要设计好
- 重复文件错误是结构化异常，不统一收口会导致页面逻辑分散
- 两种模式的结果结构不同，如果类型设计不清晰，后期很容易把页面写乱
