# RecToForm 前端 UI 样式交付文档

## 1. 文档目的

本文档用于沉淀 RecToForm 前端工作台的 UI 样式交付标准，作为后续页面重构、美化实现、联调验收和视觉一致性检查的统一依据。

交付目标不是做一个普通后台表单页，而是做一个具备“智能识别控制台”气质的单页工作台：深色、精密、清晰、可编辑、具备科技感，并能承载上传、识别、预览、导出的一体化流程。

## 2. 交付范围

本次 UI 样式交付覆盖以下区域：

- 页面整体背景与品牌头部
- 中央共享工作区
- 模式切换与模板引导
- 上传区与文件列表
- 任务进度区与处理中遮罩
- 结果预览 / 编辑表格区
- 失败项区
- 下载 / 导出区
- 帮助中心固定入口
- 桌面端优先的响应式适配

本次交付不包含：

- 后端接口协议调整
- 业务状态机重写
- 新增导出协议或新的识别能力

## 3. 视觉基调

### 3.1 关键词

- 智能识别工作台
- 深色未来感
- 数据控制台
- 精密
- 清晰
- 克制
- 可持续扩展

### 3.2 视觉结论

整体采用深色工作台风格：

- 页面基底使用深蓝黑、石墨黑
- 功能高亮使用青蓝、冰蓝、荧光绿
- 背景加入弱网格、径向光晕、模糊渐变
- 面板使用半透明深色底、细描边、微发光
- 重点动作以明确主按钮突出，例如“分析填表”“下载 Excel”“导出表格”

明确避免：

- 白底通用后台风格
- 廉价紫色渐变科技风
- 满屏同质化圆角卡片

## 4. 页面布局交付标准

### 4.1 顶部品牌头部

头部必须展示：

- `RecToForm`
- 英文或中文副标题
- 当前处理模式
- 系统状态位或任务状态位

头部应具备控制台抬头感，而不是普通标题栏。可采用横向渐变描边、弱光线、状态徽标等方式强化品牌感。

### 4.2 中央共享工作区

中央共享工作区是页面最核心的交付区域，必须统一承载以下四个阶段：

1. 上传文件
2. 选择模板
3. 分析处理中
4. 结果预览与编辑

交付要求：

- 各阶段在同一主舞台区域切换
- 容器尺寸尽量稳定，避免页面频繁跳动
- 切换过程可使用淡入、位移、轻模糊等动效
- 让用户形成“在同一工作台中推进任务”的连续感

### 4.3 下方辅助区

共享工作区下方应保留两类辅助区域：

- 失败项区
- 下载 / 导出区

二者应明显次于主舞台，但仍保持高可读性和清晰边界。

### 4.4 左下角帮助中心

页面左下角保留固定“帮助中心”入口。

要求：

- 常驻可见
- 风格低调
- 后续可挂接说明页或帮助弹层

## 5. 核心模块交付标准

### 5.1 模式切换与模板引导

模式区不应只是普通按钮或下拉框，而应具备引导性。

交付要求：

- `template` 与 `standard_edit` 以卡片式切换呈现
- 明确展示模式用途和差异
- `template` 模式下展示模板卡片和表头预览
- `standard_edit` 模式下展示全字段返回说明
- 激活态具有明显高亮描边或辉光反馈

### 5.2 上传区

上传区是工作台的第一主入口，必须承担足够强的引导职责。

交付要求：

- 支持点击上传、拖拽上传的视觉入口
- 预留选择文件夹上传的入口位
- 明确显示文件限制：图片/PDF、单文件 10MB、单次 50 个
- 文件列表展示文件名、大小、状态、警告、重复标记
- 空态清晰，已选态有层次

### 5.3 任务进度区

任务进度区需要提供明确、即时、可信的反馈。

交付要求：

- 同时显示进度条、百分比、状态文案、阶段文案
- 展示 `processed / total`、成功数、失败数
- 状态标签至少覆盖 `queued`、`running`、`succeeded`、`partially_succeeded`、`failed`
- 在“分析处理中”阶段，中央共享工作区进入雾化遮罩态，并在中心显示进度组件

### 5.4 表格结果区

表格区是页面的核心主舞台，必须承担“预览结果 + 人工校对 + 编辑导出”的视觉中心职责。

交付要求：

- 基于 `Handsontable` 实现
- `template` 模式下为只读预览
- `standard_edit` 模式下为可编辑表格
- 表头、选中态、焦点态、工具栏都要与整体工作台风格统一
- 编辑模式下明确展示“可自定义表头”的提示，不依赖 toast
- 表格容器应具备独立边界和足够面积

### 5.5 下载与导出区

下载与导出区需要形成明确的结果交付动作。

交付要求：

- `template` 模式主按钮语义为“下载 Excel”
- `standard_edit` 模式主按钮语义为“导出并下载 Excel”
- 主按钮视觉优先级高于普通交互按钮
- 导出文件名输入区、链接反馈、错误信息应有清晰层次

### 5.6 失败项区

失败项必须显式展示，不得被弱化为提示文案。

交付要求：

- 展示失败数量、失败原因数量、当前筛选数
- 支持搜索与原因筛选
- 列表项具备清晰文件名与错误信息层级
- 视觉风格偏向“异常面板”，但不压过主舞台

## 6. 视觉系统交付建议

### 6.1 颜色系统

建议至少定义以下层级：

- 页面背景色
- 面板背景色
- 高亮主色
- 辅助信息色
- 成功色
- 警告色
- 错误色
- 边框色

### 6.2 字体系统

建议区分：

- 品牌标题字体
- 正文与界面字体
- 数值与状态字体

其中状态数字、进度百分比、任务统计可适度采用偏等宽风格，增强仪表感。

### 6.3 面板系统

统一面板规范：

- 深色半透明底
- 细描边
- 控制阴影深度
- 激活态可出现青蓝色边缘辉光

### 6.4 控件系统

统一规范：

- 主按钮
- 次按钮
- 输入框
- 标签
- 状态徽标
- 工具栏按钮

所有控件需在 hover、focus、disabled、active 下保持一致的交互语言。

## 7. 动效交付标准

动效必须服务于状态反馈，而不是为了炫技。

建议交付以下动效：

- 页面加载的分区渐进出现
- 模式卡片 hover 的轻位移和描边增强
- 上传区 hover / 拖拽进入时的发光响应
- 阶段切换时的淡入或轻位移过渡
- 处理中遮罩层的轻呼吸或脉冲动画

不建议：

- 大幅旋转
- 高频闪烁
- 干扰表格阅读的持续动画

## 8. 状态覆盖清单

交付时必须覆盖：

- 初始化加载态
- 初始化失败态
- 上传空态
- 上传后待选模板态
- 模板详情加载态
- 任务进行中态
- 任务失败态
- 任务部分成功态
- 结果空态
- 导出中态
- 导出完成态

## 9. 响应式交付标准

主场景以桌面端为主，但需要保证中等屏幕可用。

要求：

- 桌面端优先使用宽屏工作台布局
- 中等屏下，上传区、进度区、辅助区允许上下堆叠
- 结果表格保持横向滚动，不强行压缩列宽
- 帮助中心入口在窄屏下可调整为更适合点击的位置

## 10. 验收标准

交付验收时，以以下结果为准：

1. 首屏明确传达“RecToForm 智能识别工作台”的品牌感
2. 页面主流程被统一收敛到中央共享工作区
3. 整体视觉符合深色、科技感、控制台式的要求
4. 上传、模板、处理中、结果四阶段切换清晰
5. `template` 与 `standard_edit` 两种模式在视觉和行为上区分明确
6. 结果表格成为页面主舞台，且 `Handsontable` 风格已融入整体设计
7. 失败项与导出区具备清晰层级和可操作性
8. 中等屏幕下页面仍可使用

## 11. 交付结论

RecToForm 的 UI 样式交付不应停留在“把现有表单换一套颜色”，而应完成从“接口调试页”到“智能识别工作台”的形态升级。

后续进入实际开发阶段时，应优先完成：

1. 页面外壳与全局视觉 token
2. 中央共享工作区与头部品牌区
3. 上传、模板、进度、结果四个核心阶段的主视觉
4. 失败项、导出区与帮助中心入口的收尾统一

以此作为 UI 表现层重构的执行基线和验收基线。
